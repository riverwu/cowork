# Cowork — 产品形态文档

## 设计前提

### 1. 不区分创建者和使用者

所有用户共享同一套界面。能力差异体现在使用深度上（随手用 → 保存 → 参数化 → 共享 → 发布），而不是不同的功能入口。

### 2. 自然语言是主要交互方式

定义 App、运行任务、调整参数、修改输出——都通过对话完成。结构化控件（表单、下拉框）作为辅助引导存在，不是主要交互。

### 3. 用户定义意图，AI 决定执行方式

用户不需要设计 Agent 编排图。用户描述目标和约束，AI 在运行时自行决定如何分解任务。运行过程透明可查看，但不是用户需要预先设计的。

### 4. 界面按需浮现，而非固定面板

不预设固定的多面板布局。AI 根据对话上下文，自动展示最相关的视图（数据表格、图表、报告预览）。用户可以钉住某个视图保持可见。

## 整体信息架构

```
Cowork
├── Home（首页）                  ← 行动优先的默认页面
│   ├── Command Bar（命令栏）——核心行动入口
│   ├── Quick Actions（快捷行动——运行 App、创建 App）
│   ├── Incoming（待处理——来自 Channel 的输入和待审阅）
│   └── Recent（最近活动——运行结果、产出物）
│
├── App Detail（应用详情）    ← 从 Home 点击 App 进入
│   ├── Conversation（对话式运行/调整）
│   ├── Run History（运行历史）
│   └── Settings（App 配置）
│
├── Knowledge（知识库）            ← 用户的资料库，AI 工作时自动引用
│   ├── My Documents（我的文档）
│   ├── Connected Sources（已连接的数据源）
│   └── Cowork Outputs（Cowork 产出的内容）
│
├── Explore（发现）               ← 发现组织内共享的 App
│
├── Activity（运行监控）           ← 所有 App 的运行状态
│
├── Channels（通道）              ← 管理输入输出通道（IM、邮件、Webhook）
│
└── Settings（全局设置）
    ├── Skills（技能管理——查看可用 Skill、管理员控制权限）
    └── Team & Permissions（团队与权限）
```

**Knowledge 页面的定位：** 用户的资料库——"我的东西都在这里"。用户在这里添加文档目录、连接数据源、控制哪些资料 AI 可以用。底层的索引和 AI 理解机制完全隐藏。

**定位：** 行动优先的落地页。用户打开 Cowork 就是来干活的——立即开始任务、运行 App、处理待办。不是仪表盘，是工作台。

**Command Bar — 最核心的行动入口：**
- 位于页面顶部最显眼位置，始终可见
- 这是 Cowork 的统一入口——无论是即时执行新任务，还是运行已有 App，都从这里开始
- 用户用自然语言描述想做的事，系统立即执行
- 如果描述匹配已有 App，系统提示"你有一个类似的 App'周度销售异常分析'，要用它来运行吗？"
- 示例输入：
  - "帮我看看上周 Shopify 各品类的销售变化，有没有异常的"
  - "做一个 App，每天早上检查客服邮箱里的紧急投诉并推送到飞书群"
  - "跑一下周度销售分析，这次重点看户外装备品类"

**我的 App — 一键运行：**
- 紧挨 Command Bar 下方，最常用的 App 一目了然
- 每张卡片直接可点击运行（▶ 图标），不需要进入详情再运行
- 显示最近运行状态，快速判断是否正常
- 末尾的"+ 新建 App"卡片引导创建

**关键交互：**
- 执行过程中，AI 的步骤透明展示（可观测性），但用户不需要预先定义这些步骤
- 中间结果以按需浮现的视图展示——数据表格、图表、文本分析
- 用户可以在结果基础上继续对话追问
- "发送到..."按钮——任务完成后直接通过 Channel 发布结果（发邮件、推飞书群、归档到 Confluence）
- 对话变复杂时，界面自动扩展为更宽敞的工作视图
- 任何一次执行都可以点击"保存为 App" → 系统从对话历史中提取意图和约束

**"待处理"区域：**
- 聚合所有需要用户行动的事项：
  - 来自 Channel 的输入（飞书群里的 @mention、邮件中的请求）
  - App 运行结果待审阅
  - 运行失败需处理
- 每条带明确的行动按钮，一键进入处理
- 无待办时此区域收起

### 页面 2：Instant Session（即时对话工作区）

**定位：** 从 Command Bar 发起的即时任务，如果需要多轮深入探索，会自然扩展为一个对话工作区。不是独立的"页面"，而是 Command Bar 交互的自然延伸。

└─────────────────────────────────────────────────────┘

**对话区（左栏）— 主要交互区域：**
- 用户通过自然语言持续指挥和追问
- AI 展示执行过程、中间发现、最终产出
- 对话中提到的数据和产出自动在右侧视图区呈现

**视图区（右栏）— 按需浮现：**
- 不是固定的面板，而是随对话内容动态出现的视图
- AI 生成数据表格时 → 表格视图浮现
- AI 生成报告时 → 报告预览视图浮现
- 每个视图可以：
  - **钉住**（📌）：保持可见，不随对话滚动消失
  - **全屏**：展开为大视图（查看复杂图表或长报告）
  - **关闭**：不需要时收起
- 如果没有视图需要展示，右栏收起，对话区占满全宽

**与 Command Bar 的关系：**
- 简单任务在 Command Bar 内联完成（不跳转页面）
- 当对话超过 2-3 轮或产出物需要详细查看时，自动扩展为 Instant Session 全页面
- 扩展是无缝的——对话历史完整保留

**Knowledge 在 Instant Session 中的体现：**
- AI 在执行任务时自动引用相关 Knowledge，在对话中自然提及（"我参考了你 Q1 的分析报告的格式"）
- 底部状态栏显示 AI 引用的知识来源（"Knowledge: Q1流失分析.docx · CRM 客户记录 3条"）
- 用户可以在对话中纠正（"别用那个格式，用这个模板"），AI 立即调整并记住这个偏好
- 任务完成后，本次产出自动被 AI 理解并纳入 Knowledge，为未来任务积累上下文

**关键交互：**

| 操作 | 行为 |
|------|------|
| 在对话区继续追问 | AI 执行后续任务，视图区更新 |
| 在视图区选中数据 | 对话区自动关联上下文，可针对选中内容提问 |
| 点击"保存为 App" | 系统从对话历史提取意图和约束，生成 App |
| 点击"发送邮件" | 将当前产出物通过邮件发送 |
| 关闭 Session | 执行记录保存在"最近产出"中，产出物自动纳入 Knowledge |

**对话栏 — App 的万能操作入口：**

所有操作都可以通过自然语言完成：

| 用户说 | 系统做 |
|--------|--------|
| "跑一下，这次只看户外装备" | 以指定参数立即运行 |
| "把判断标准改一下，下降超过20%才算异常" | 修改 App 定义中的质量标准 |
| "加上运营日志作为数据源" | 添加数据源 |
| "以后如果有品类下降超过30%，额外抄送给VP" | 修改信任边界和输出通道 |
| "看看上次的报告" | 展示最近一次运行的产出物 |
| "改成每天运行" | 修改调度频率 |
| "分享给市场部" | 修改可见范围 |
| "上次报告里归因分析不太准，季节性判断要参考去年同期" | 优化质量标准 |

**快捷按钮：** 对话栏下方提供常用操作的快捷入口（立即运行、查看上次结果、调整 App），但这些只是对话的快捷方式，点击后等效于在对话中输入对应的指令。

**App 定义区：**
- 展示当前 App 的完整定义——目标、数据源、质量标准、输出要求、信任边界、参数、分发、调度
- 所有字段都是自然语言文本，不是结构化表单
- 可以直接点击某个字段编辑（适合微调），也可以在对话栏中描述修改意图（适合复杂调整）
- 每次修改后 App 版本自动递增

**运行历史区：**
- 每次运行的时间、状态、产出物入口
- "查看过程"：展示 AI 在该次运行中的执行步骤（可观测性），包括访问了哪些数据、做了什么判断、中间结果是什么
- 可以对比不同次运行的结果差异

## 导航结构

**统一导航（左侧主导航）：** 所有用户共享同一套导航。

```
🏠 Home           ← 行动优先（Command Bar、我的 App、待处理）
🧠 Knowledge      ← 你的资料库（文档、数据源、历史产出）
🔗 Channels       ← 输入输出通道（飞书、邮件、Webhook 等）
🔍 Explore        ← 发现组织内共享的 App
📊 Activity       ← 运行监控（有定时任务时更常用）
⚙️ Settings       ← Skill 管理、团队管理
```

**没有"创建"入口：** 创建行为不需要独立的导航入口。所有创建都从日常使用中自然发生：
- 在 Command Bar 执行任务后 → "保存为 App"
- 在 App Detail 对话中 → 随时调整 App 定义
- 在 Explore 中看到他人 App → "复制并修改"

**没有"上传文档"入口：** 文档不是"上传到 Cowork"的，而是用户授权 Cowork 访问自己的工作目录/云盘/协作平台。AI 持续索引和理解，用户无需手动管理。

## 渐进式固化的 UI 表现

固化不是一个独立的流程，而是一系列轻量操作：

```
Level 0: 即时执行
  用户在 Command Bar 执行任务
  → 结果展示 + [保存为 App] 按钮

Level 1: 保存
  用户点击"保存为 App"
  → 系统自动提取意图和约束，生成 App 草稿
  → 用户在 App Detail 中可以直接编辑或通过对话调整

Level 2: 参数化
  用户在 App Detail 对话中说"让日期范围可以每次调整"
  → 系统标记 date_range 为参数，下次运行时可指定

Level 3: 共享
  用户说"把这个分享给销售部"
  → App 出现在销售部同事的 Explore 页面

Level 4: 配置触发和输出通道
  用户说"每周一早上9点自动跑一次，报告发到飞书 #sales-ops 和 sales-team@"
  → 系统配置定时触发和输出 Channel
  用户说"如果飞书群里有人 @销售助手 问销售数据的问题，也用这个 App 回答"
  → 系统配置飞书 Channel 作为输入触发
```

每一步都是在 App Detail 的对话栏中用一句话完成的，不需要跳转到不同的页面或模式。

## 响应式设计要求

- **桌面优先**：Instant Session 的对话+视图布局设计为桌面宽屏体验（最小 1280px）
- **移动端完整适配**：
  - Home：Command Bar + 待办 + App 卡片，完整可用
  - App Detail：对话式交互在移动端天然适配
  - 产出物查看：报告、邮件预览在移动端完整可读
  - 审阅操作：移动端可完成审阅和批准（高频移动场景）
- **移动端视图区**：在 Instant Session 中，视图区从右侧并排改为底部 sheet（可上滑展开）
