# Cowork — 核心用户流程

## 流程总览

Cowork 的用户旅程从首次体验开始，然后是一个连续的光谱：

```
Flow 0           Flow 1           Flow 2           Flow 3          Flow 4          Flow 5          Flow 6
首次体验     →    即时执行     →    保存为 App   →    调整优化    →   共享发布    →   持续迭代    →   使用他人 App
(扫描/setup文档)  (Command Bar)    (一键保存)       (对话调整)      (对话配置)      (反馈驱动)      (发现使用)
```

Flow 0 是首次体验——不是独立的"设置"步骤，而是直接带用户进入第一个有价值的任务。设置在使用过程中按需发生。

---

## Flow 0：首次体验

**核心原则：不把"设置"和"使用"分开。第一次交互就是一个真实任务，设置在任务中自然发生。**

有两条路径进入，取决于用户的起点：

### 路径 A：个人快速开始（最常见）

**场景：** 张三自己下载了 Cowork，想试试看。

```
┌─────────────────────────────────────────┐
│ 0:00 首次打开 Cowork                     │
│                                         │
│ AI: "我是 Cowork。你指一个工作文件夹给我，│
│   我看看你在做什么，然后帮你干活。"        │
│   [选择文件夹]                           │
│                                         │
│ 0:10 张三选了 ~/Documents/Work           │
│                                         │
│ 0:30 AI 快速扫描文件名和修改时间          │
│   （秒级完成，不是深度索引）               │
│                                         │
│ 0:45 AI: "我看了一下，你最近在做这些：    │
│                                         │
│   📄 Q1 客户流失分析.docx                │
│      3 周前更新                          │
│   📊 周度销售数据汇总.xlsx               │
│      每周更新，上次是周一                  │
│   📄 Project Phoenix 方案 v2.pdf         │
│      上周更新                            │
│                                         │
│   有什么我现在就能帮你做的？比如：         │
│   · Q1 的流失分析做完了，要做 Q2 的吗？   │
│   · 这周的销售汇总还没做，要现在做吗？"    │
│                                         │
│ 1:00 张三: "帮我做 Q2 的客户流失分析"     │
│                                         │
│ → AI: "好的。我参考你 Q1 的报告格式。     │
│   不过我需要客户数据——你们用什么 CRM？"   │
│                                         │
│ → 张三: "Salesforce"                     │
│ → [一键授权 Salesforce]                  │
│                                         │
│ → AI 开始工作：                          │
│   · 参考 Q1 报告的格式和结构              │
│   · 从 Salesforce 拉取 Q2 客户数据        │
│   · 生成 Q2 流失分析报告                  │
│                                         │
│ 2:00 AI: "报告初稿好了。我参考了你 Q1     │
│   报告的结构，从 Salesforce 拉了 Q2 数据。│
│   你看看有什么要改的。"                    │
│                                         │
│   [查看报告]  [发送到...]  [保存为 App]    │
│                                         │
└─────────────────────────────────────────┘
```

**为什么这样设计：**

1. **一个动作启动一切** — 用户只做了一件事（选了一个文件夹），AI 就已经知道他在做什么工作。不需要填表、选数据源、配置任何东西。

2. **AI 的建议是具体的** — 不是"你可以用我来分析数据"（泛泛的功能介绍），而是"你 Q1 的流失分析做完了，要做 Q2 的吗"（基于用户当前工作的具体建议）。这是魔法时刻——用户立刻感受到"这不是又一个通用 AI 工具"。

3. **数据源按需连接** — 不是提前要求连好所有数据源，而是做 Q2 分析需要 CRM 时才问。这时用户有动力授权——因为他正在做一个具体任务。

4. **快速扫描 vs 深度索引** — 第一次只扫文件名和修改时间（秒级），足以理解用户在做什么。深度索引在后台异步进行，不阻塞第一次体验。

### 路径 B：Setup 文档引导（团队/企业场景）

**场景：** 张三的公司已经在用 Cowork，IT 部门准备了一份 setup 文档（或公司内部 wiki 上有一个 setup 页面），新员工/新用户照着来就行。

```
┌─────────────────────────────────────────┐
│ 0:00 首次打开 Cowork                     │
│                                         │
│ AI: "我是 Cowork。你指一个工作文件夹给我，│
│   我看看你在做什么，然后帮你干活。         │
│   如果你有 setup 文档，也可以直接给我。"   │
│   [选择文件夹]  [导入 setup 文档]         │
│                                         │
│ 0:05 张三点击"导入 setup 文档"            │
│   → 选了公司内网的 Cowork Setup Guide     │
│     （或者同事发给他的 setup.md 文件）      │
│                                         │
│ 0:10 AI 读取文档，理解内容：              │
│                                         │
│   "我读完了。这份文档描述了你们销售部的     │
│    工作环境，我来帮你配置好：              │
│                                         │
│    ✅ 连接 Salesforce (CRM)              │
│       → [点击授权]                       │
│    ✅ 连接 Shopify (电商数据)             │
│       → [点击授权]                       │
│    ✅ 连接 Gmail 客服邮箱                 │
│       → [点击授权]                       │
│    ✅ 连接飞书 #sales-ops 群 (通道)       │
│       → [点击授权]                       │
│    ✅ 添加共享 App: 周度销售异常分析       │
│    ✅ 添加共享 App: 客户满意度追踪         │
│                                         │
│    需要你逐个授权一下。"                   │
│                                         │
│ 0:30 张三逐个点击授权                     │
│                                         │
│ 1:00 AI: "都配好了。你已经有 2 个可用的    │
│   App 了。你们团队的'周度销售异常分析'     │
│   每周一 9:00 自动运行。                  │
│                                         │
│   要不要现在试试？跑一下这周的销售分析？"  │
│                                         │
│ → 张三: "好"                             │
│ → App 立即运行，产出结果                  │
│                                         │
└─────────────────────────────────────────┘
```

**Setup 文档可以是什么：**

- **公司 IT 准备的配置指南** — 一个 wiki 页面或 PDF，描述"销售部需要连哪些系统、用哪些共享 App、飞书群是哪个"
- **同事导出的配置** — 在 Cowork 中点击"导出我的配置"，生成一份 setup 文档，发给新同事
- **团队约定文档** — 不是专门为 Cowork 写的，而是团队本来就有的工作规范文档（"我们用 Salesforce 管客户、Shopify 是电商平台、周报发到 #sales-ops"），AI 能从中理解需要连什么

**Setup 文档不需要是结构化的配置文件。** 它可以就是一段自然语言：

```markdown
## 销售部工作环境

我们的 CRM 是 Salesforce，电商平台是 Shopify。
客服邮件在 Gmail 的 support@ 邮箱里。
团队沟通用飞书，主要群是 #sales-ops。
每周一早上需要做销售异常分析，结果发到飞书群和 sales-team@ 邮箱。
客户满意度数据每月从 Salesforce 导出分析一次。
```

AI 读完这段话，就知道要连什么数据源、配什么 Channel、甚至建议创建什么 App。

**为什么这样设计：**

1. **一份文档配好一切** — 不需要用户理解 Cowork 的概念（什么是 Knowledge、Channel、Skill），只需要给 AI 一份描述工作环境的文档。
2. **极大降低团队推广门槛** — 第一个人花时间配置好，导出 setup 文档，后面的人 1 分钟就能配完。
3. **利用已有资产** — 很多团队已经有工作规范文档、onboarding 文档，不需要专门为 Cowork 写配置。
4. **AI 做翻译层** — 自然语言的 setup 文档 → AI 翻译为具体的系统配置（连哪个 Salesforce 实例、监听哪个邮箱等）。用户只需逐个授权确认。

### 两条路径的汇合

无论走哪条路径，用户在 2 分钟内都到达了同一个状态：
- 有了第一批 Knowledge（工作文件 + 至少一个数据源）
- 完成了第一个有价值的任务
- 感受到了"这个工具懂我的工作"

之后进入 Flow 1（即时执行），持续使用。更多的知识库内容和 Channel 在后续使用中按需添加。

### 导出 Setup 文档（反向流程）

已经用了一段时间的用户，可以在 Settings 中"导出 setup 文档"：

```
┌─────────────────────────────────────────┐
│ 张三在 Settings 中点击"导出配置"          │
│                                         │
│ → 系统生成一份 setup 文档，包含：         │
│   · 已连接的数据源列表和用途说明           │
│   · 已配置的 Channel                     │
│   · 已共享的 App（名称和简介）             │
│   · 工作流程的自然语言描述                 │
│                                         │
│ → 张三把这份文档发给新同事李四             │
│ → 李四首次打开 Cowork 时导入              │
│ → 1 分钟配好，立即可用                    │
└─────────────────────────────────────────┘
```

## Flow 1：即时执行

**场景：** 销售运营负责人张三想看看上周的销售情况。张三已使用 Cowork 一段时间，知识库中有过去的工作文档和连接的数据源。

```
┌─────────────────────────────────────────┐
│ 张三在 Home 的 Command Bar 输入：        │
│ "帮我看看上周 Shopify 各品类的销售变化，  │
│  有没有异常的"                           │
│                                         │
│ → AI 自动执行：                          │
│   1. 从 Knowledge 找到张三上次的销售分析  │
│      → 知道他关注的维度和报告格式         │
│   2. 连接 Shopify（已授权的数据源）       │
│   3. 获取上周订单数据                     │
│   4. 按品类汇总，计算周环比               │
│   5. 识别异常品类                         │
│                                         │
│ → 执行过程透明展示，每一步可查看          │
│ → AI: "基于上周数据分析，发现2个品类异常。│
│   我参考了你之前的分析格式。"             │
│ → 结果内联呈现：品类环比变化表格          │
│   户外装备 -23% ⚠️ | 电子配件 -18% ⚠️   │
│                                         │
│ → 张三继续追问：                          │
│   "看看这两个品类的客诉有没有共性问题"     │
│                                         │
│ → AI 从 Gmail 客服邮箱拉取邮件并分析      │
│ → 对话自动扩展为 Instant Session          │
│   左侧对话、右侧浮现数据表格和分析卡片    │
│                                         │
│ → 张三：                                 │
│   "基于以上分析生成一份简报给管理层，      │
│    结论先行"                              │
│                                         │
│ → AI 生成报告，右侧浮现报告预览           │
│ → 张三看了下，修改了一处措辞              │
│ → 点击"发送邮件"，选择收件人，发送         │
│                                         │
│ 任务完成。底部显示：                      │
│ [保存为 App]  [关闭]                      │
└─────────────────────────────────────────┘
```

**系统后台行为：**
- 全程记录张三的意图（分析销售异常）、使用的数据源（Shopify + Gmail）、质量标准（发现异常品类、交叉分析客诉）、输出要求（管理层简报、结论先行）
- 这些记录不需要张三显式操作，自动从对话中提取
- **Knowledge 自动参与**：如果张三之前做过类似的销售分析，AI 会自动参考之前的格式和分析角度，在对话中提及"我参考了你上次的分析报告格式"。张三不需要指定这些，AI 自动关联
- **产出物自动回流**：本次生成的分析报告被 AI 理解并纳入 Knowledge。下次张三做类似分析时，AI 不仅知道格式偏好，还知道上次的分析结论（可作为同比参考）

**此刻张三可以选择：**
- 关闭——这就是一次性任务，结果已经发出（但产出物已自动纳入 Knowledge）
- 保存为 App——进入 Flow 2

## Flow 2：保存为 App

**场景：** 张三觉得这个分析以后每周都要做，决定保存。

```
┌─────────────────────────────────────────┐
│ 张三点击"保存为 App"                      │
│                                         │
│ → 系统从对话历史中自动提取：              │
│                                         │
│   App 名称: 销售异常分析（自动生成，可改）│
│                                         │
│   目标: 分析各品类销售异常并生成简报      │
│                                         │
│   数据源: Shopify 订单 · Gmail 客服邮箱   │
│                                         │
│   质量标准:                              │
│   · 按品类汇总，识别环比变化显著的品类     │
│   · 交叉对比客诉邮件寻找共性问题          │
│                                         │
│   输出: 面向管理层的简报，结论先行         │
│                                         │
│   参数: date_range (默认: 上周)           │
│                                         │
│ → 张三浏览了一下，觉得大致对             │
│ → 点击"保存"                             │
│ → App 出现在 Home 的"我的 App"中          │
│                                         │
│ 下次想运行时，可以：                      │
│ · 点击 App 卡片 → 在 App Detail 中    │
│   点击"立即运行"                          │
│ · 在 Command Bar 说"跑一下销售异常分析"   │
└─────────────────────────────────────────┘
```

**关键体验：** 保存不需要填写复杂的配置表单。系统自动从对话中提取所有必要信息，用户只需确认。整个保存过程在 3 秒内完成。

## Flow 3：调整优化

**场景：** 张三运行了几次后，想优化 App。

```
场景 A — 运行时用自然语言调参
┌─────────────────────────────────────────┐
│ 张三打开"销售异常分析"的 App Detail  │
│ → 在对话栏说：                           │
│   "跑一下，这次只看户外装备品类"          │
│ → AI 以 product_line=户外装备 运行       │
│ → 结果正常                               │
└─────────────────────────────────────────┘

场景 B — 优化质量标准
┌─────────────────────────────────────────┐
│ 张三发现报告中的季节性判断不准            │
│ → 在对话栏说：                           │
│   "判断季节性的时候应该参考去年同期数据，  │
│    不要只凭品类名称推测"                  │
│ → 系统更新质量标准，并自动添加去年同期     │
│   Shopify 数据作为数据源                  │
│ → App 版本更新为 v1.1                    │
│                                         │
│ → 张三："测试一下上周的数据"              │
│ → AI 用上周数据运行更新后的 App           │
│ → 结果中冬季服装的下降被正确标记为         │
│   季节性因素，不再列为异常                 │
│ → 张三满意                               │
└─────────────────────────────────────────┘

场景 C — 系统建议优化
┌─────────────────────────────────────────┐
│ App 运行了 3 次，张三每次都在报告的       │
│ "建议行动"部分手动补充具体负责人           │
│                                         │
│ → 系统在 App Detail 中提示：        │
│   "你经常在建议行动中补充负责人。          │
│    要不要在 App 中加上品类-负责人的对应关系，│
│    让报告自动带上负责人信息？"             │
│ → 张三确认                               │
│ → 系统更新 App 定义                        │
│ → 下次运行自动包含负责人信息               │
└─────────────────────────────────────────┘

场景 D — 添加信任边界
┌─────────────────────────────────────────┐
│ 张三在对话栏说：                          │
│ "如果有品类下降超过30%，不要自动发报告，   │
│  让我先看一眼"                            │
│ → 系统在 App 中添加信任边界规则：           │
│   异常幅度 ≥ 30% → 暂停，通知张三审阅     │
│ → App 版本更新                            │
└─────────────────────────────────────────┘
```

**关键体验：** 所有调整都在同一个 App Detail 页面的对话栏中完成。不需要切换到"编辑模式"或打开"Workflow Editor"。说一句话就改了。

## Flow 4：共享发布

**场景：** 张三的 App 已经稳定运行了几周，想分享给团队。

```
渐进式共享 — 每一步都是一句话
┌─────────────────────────────────────────┐
│                                         │
│ 第一步：分享                             │
│ 张三在对话栏说：                          │
│   "把这个 App 分享给销售部"                │
│ → 系统设置可见范围：销售部全员             │
│ → App 出现在销售部同事的 Explore 页面     │
│ → 同事可以添加到自己的 App 列表中使用       │
│                                         │
│ 第二步：配置输出通道                      │
│ 张三："运行完自动发邮件给 sales-team@，    │
│   同时推送到飞书 #sales-ops 群"           │
│ → 系统配置输出 Channel                    │
│                                         │
│ 第三步：配置触发方式                      │
│ 张三："每周一早上9点自动运行"              │
│ → 系统配置定时触发                        │
│ 张三："如果飞书群里有人 @销售助手 问       │
│   销售数据，也用这个 App 回答"            │
│ → 系统配置飞书 Channel 作为输入触发        │
│                                         │
│ 第四步（可选）：正式发布                   │
│ 张三："给这个 App 起个正式名字，            │
│   叫'周度销售异常分析'"                   │
│ → 系统更新 App 名称                        │
│ → 销售部同事收到通知：                    │
│   "张三发布了'周度销售异常分析'"           │
│                                         │
└─────────────────────────────────────────┘
```

**关键体验：** 发布不是一个独立的步骤，而是共享 + 调度 + 分发的组合。每一步都是可选的，每一步都是一句话。

## Flow 4.5：Channel 输入触发

**场景：** App 已经配置了 Channel 输入，外部事件触发了运行。

```
场景 A — 飞书群 @mention 自动处理
┌─────────────────────────────────────────┐
│ 同事王五在飞书 #sales-ops 群里说：       │
│   "@销售助手 上周户外装备品类怎么样？"    │
│                                         │
│ → Cowork 接收到 Channel 输入             │
│ → 匹配到"周度销售异常分析"App            │
│ → AI 解析问题，自动填入参数：             │
│   product_line=户外装备, date_range=上周  │
│ → App 自动运行                           │
│                                         │
│ → 信任边界检查：                          │
│   App 设置为"@mention 触发可自动回复"     │
│   → 直接回复                             │
│                                         │
│ → 飞书群里出现回复：                      │
│   "上周户外装备品类销售额环比下降 23%，    │
│    主要原因是发货延迟导致客诉增加。        │
│    详细报告: [查看完整分析]"              │
│                                         │
│ 王五不需要知道 Cowork 的存在              │
│ 他只是在飞书群里问了一个问题              │
└─────────────────────────────────────────┘

场景 B — Channel 输入进入"待处理"
┌─────────────────────────────────────────┐
│ 同事赵六在飞书群里问了一个复杂问题：      │
│   "@销售助手 对比一下上月和去年同期的      │
│    户外装备品类表现，分析长期趋势"         │

│ → Cowork 接收到 Channel 输入             │
│ → AI 判断：这个问题超出了 App 的         │
│   常规参数范围，需要人确认                │
│ → 不自动回复，而是出现在张三的            │
│   首页"待处理"列表中：                    │
│                                         │
│   📨 飞书 @销售助手 — 赵六问"对比上月     │
│      和去年同期..." · [去处理]            │
│                                         │
│ → 张三点击"去处理"                        │
│ → 进入一个 Instant Session，上下文已预填：│
│   AI: "赵六在飞书群里问了这个问题。       │
│   我可以用'周度销售异常分析'App 来处理，   │
│   但需要调整参数为长期对比模式。           │
│   要我运行吗？还是你想手动调整？"          │

│ → 张三: "运行吧，结果直接回复到群里"      │
│ → 运行完成，结果通过飞书 Channel 回复     │
└─────────────────────────────────────────┘

场景 C — 定时触发
┌─────────────────────────────────────────┐
│ 周一 09:00，定时触发 App 运行             │
│ → "周度销售异常分析"自动运行              │
│ → 信任边界检查：异常幅度 18% < 30%       │
│   → 自动通过                             │
│ → 结果通过 Channel 输出：                 │
│   · 邮件发送至 sales-team@               │
│   · 飞书 #sales-ops 群推送摘要           │
│ → 产出物自动回流到 Knowledge              │
│                                         │
│ 张三不需要做任何事                        │
│ 打开 Home 时在"最近活动"中看到运行记录    │
└─────────────────────────────────────────┘
```

**Channel 输入的处理逻辑：**
- AI 自动判断输入是否在 App 的参数范围内
- 范围内 + 信任边界允许自动回复 → 直接处理并通过 Channel 回复
- 范围外或需要人确认 → 进入 App 创建者的"待处理"列表
- 用户点击"去处理"→ 打开一个预填上下文的 Instant Session，用户决定如何回应

## 关键状态转换图

```
  ┌──────────────────────────────────────────────────────────┐
  │                   Knowledge（持续积累）                    │
  │  个人文档 + 连接的数据源 + 每次任务的产出 → AI 持续理解    │
  └────────────────────────┬─────────────────────────────────┘
                           │ AI 在每个环节自动调用
                           ▼
                    ┌──────────────┐
                    │   Instant     │
                    │   Session     │──── 产出物回流 ──→ Knowledge
                    │  (即时执行)    │
                    └──────┬───────┘
                           │ "保存为 App"（一键，3秒）
                           ▼
                    ┌──────────────┐
                    │   App    │
                    │   (个人 App)   │──── 运行历史回流 → Knowledge
                    └──────┬───────┘
                           │ "分享给销售部"（一句话）
                           ▼
                    ┌──────────────┐
                    │   Shared      │
                    │   App    │
                    │  (共享 App)    │
                    └──────┬───────┘
                           │ "每周一自动运行"（一句话）
                           ▼
                    ┌──────────────┐
                    │   Published   │
                    │   App         │──── 每次运行回流 → Knowledge
                    │  (已发布 App)  │
                    └──────┬───────┘
                           │
               ┌───────────┼───────────┐
               ▼           │           ▼
        ┌────────────┐     │    ┌────────────┐
        │  Channel    │     │    │  Channel    │
        │  (输出)     │     │    │  (输入)     │
        │ 邮件/飞书/  │     │    │ @mention/   │
        │ Confluence  │     │    │ Webhook/    │
        └────────────┘     │    │ 定时触发    │
                           │    └────────────┘
                           │
                           │◀── 持续迭代 ──┐
                           │   (对话调整    │
                           │    反馈驱动    │
                           └───────────────┘
```

每一步状态转换都是一个轻量操作（一键或一句话），不需要切换页面或学习新界面。Knowledge 贯穿整个过程——AI 在每个环节自动调用已有知识，每个环节的产出又自动回流为新的知识。
