# Cowork — 产品定义文档

## 一句话定义

Cowork 是一个让知识工作者在 AI 持续理解其工作上下文的基础上，通过定义意图和约束来完成任务、沉淀为可复用的 App，并在组织内共享的平台。

---

## 1. 产品定位

### 我们解决什么问题

知识工作者每天处理来自多个数据源的信息（邮件、IM、日志、Dashboard、销售数据），并产出分析报告、改进计划、邮件等。这些工作中有两类资产长期被浪费：

1. **可复用的工作模式** — 存在于个人经验中，无法固化、无法共享、每次从头做
2. **积累的工作上下文** — 过去写的报告、分析、方案散落在各处，每次启动新任务都要重新收集背景信息

**Cowork 让 AI 持续理解用户的工作上下文，在每次任务中自动调用相关的知识和数据；同时将工作方法沉淀为可复用的 App，以极低门槛共享给组织。**

核心价值是双重的：
- **短期**：AI 了解你的工作，每次任务都站在已有积累之上，而不是从零开始
- **长期**：工作模式被固化为可复用的 App，个人能力变成组织能力

### 我们不做什么

- 不做通用聊天助手（ChatGPT 已经做了）
- 不做绑定单一生态的 AI 功能（Notion AI / Copilot 已经做了）
- 不做无智能的自动化流转（Zapier / RPA 已经做了）
- 不做面向程序员的开发工具（Claude Code 已经做了）
- 不做文档管理系统（Notion / Google Drive 已经做了）— 我们不存储和组织文件，而是让 AI 自动从用户已有的文件和系统中找到相关资料来用

### 核心差异化

| 维度 | 现有工具 | Cowork |
|------|---------|--------|
| 上下文 | 每次对话从零开始 | AI 持续理解用户的工作上下文 |
| 产出物 | 一次性结果（回答、文档） | 可复用的 App + 自动积累的知识 |
| 定义方式 | 脚本/低代码/拖拽/Agent 编排 | 意图 + 约束（自然语言） |
| 判断力 | 确定性逻辑 (if/else) | AI 运行时理解与判断 |
| 运行模式 | 会话式（启动→结束） | 持续运行（调度、触发、监控） |
| 复用路径 | 分享 prompt / 模板 | 渐进式固化：随手用 → 保存 → 共享 → 发布 |
| 数据利用 | 手动指定数据来源 | 自动关联相关知识和数据 |

## 2. 目标用户

### 不做硬性角色区分

传统思路会把用户分为"创建者"和"使用者"，但 AI 能力的发展正在模糊这个边界。在 Cowork 中，所有用户都在同一个连续光谱上：

```
随手用一次 → 保存为 App → 参数化 → 分享给同事 → 发布为组织工具
```

每个人都可能在不同时刻处于光谱的不同位置。不需要"切换到创建模式"，每一步都是从上一步的自然延伸。

### 用户画像

**高能力工作者（光谱的右侧，频繁使用）**
- 业务分析师、运营负责人、项目经理、高级产品经理
- 日常处理多数据源信息，产出分析和决策
- 有较强的逻辑能力和业务判断力
- 在组织中是"问题解决者"角色
- 核心诉求：把自己的工作方法变成可复用的 App，放大个人影响力

**一般工作者（光谱的左侧，日常使用）**
- 一线业务人员、初级分析师、行政人员
- 使用他人创建的 App，或用自然语言直接完成简单任务
- 核心诉求：简单地完成工作，不需要学习新技能

**关键设计原则：** 两类用户使用同一套界面、同一个入口。能力差异体现在使用深度上，不体现在功能入口上。

### 使用场景举例

- 销售运营负责人：每周从电商平台、客服邮箱、运营日志中提取异常，生成分析报告发给管理层 → 固化为每周自动运行的 App
- 项目经理：从 Jira、Confluence、Slack 中汇总项目进展，生成周报并识别风险 → 分享给其他 PM 复用
- 市场分析师：从多个数据源交叉分析 campaign 效果，生成优化建议 → 发布为组织级 App
- 一线销售：在 Command Bar 输入"帮我查一下客户 A 最近的订单和客诉情况"→ 直接获得结果

## 3. 核心概念模型

### 3.1 Knowledge（知识库）— 核心概念

Knowledge 是用户在 Cowork 中的资料库。首次使用时用户只需指一个工作文件夹，AI 就能快速了解用户在做什么工作；更多数据源在使用过程中按需连接。AI 干活时会自动从知识库中找到相关的来用。

**知识库里有三类内容：**
- **我的文档** — 用户授权的工作目录（本地文件夹、云盘）
- **已连接的数据源** — CRM、Confluence、邮箱等外部系统
- **Cowork 的产出** — 每次任务生成的报告、分析等，自动保存

**用户的操作是具体的、简单的：**
- 添加/移除文档目录
- 连接/断开数据源
- 浏览文件列表，排除不想被使用的文件
- 设置隐私级别（"这个目录仅个人可用"）

**AI 的工作是自动的、隐藏的：**

用户说"帮我写一份 Q2 的客户流失分析报告"，AI 自动从知识库中找到相关资料来用——Q1 的同类报告（参考格式）、过去的客户分析（参考上下文）、CRM 的流失数据。用户在执行过程中能看到 AI 参考了什么，但不需要提前指定。如果 AI 用的资料不对，用户直接说"别用那个，用这个"就行。

底层的索引、语义理解、关联推理、RAG 检索等机制完全不暴露给用户。

**隐私控制：**
- 标记为"仅个人可用"的目录：AI 在执行共享 App 时不会引用
- 共享 App 时：App 定义共享，创建者的个人资料不共享
- 排除的文件：AI 完全不使用

### 3.2 Skill（技能）

Skill 是 AI 的能力单元——通用的、可复用的、跨 App 的。

**内置 Skill 示例：**
- 浏览网页（browser-use）
- 读取和分析表格数据
- 从 CRM / ERP 拉取数据
- 从邮箱搜索和分析邮件
- 生成结构化报告
- 发送邮件 / Slack 消息
- 读取和理解 PDF 文档
- 代码执行（数据处理、计算）

**用户不需要选择或编排 Skill。** AI 根据 App 的意图和约束，在运行时自主决定使用哪些 Skill。执行日志中用户能看到用了哪些 Skill（透明度），但不需要提前指定。

**Skill 的扩展性：**
- 平台内置核心 Skill
- 组织管理员可以控制哪些 Skill 可用（比如禁用外部网页访问）
- 未来可支持第三方 Skill（比如对接特定行业系统的 Skill）
- 新增 Skill 后，所有现有 App 自动获得新能力——因为 App 只定义意图，AI 运行时自主选择 Skill

**Skill 在产品中的位置：** 不是用户日常操作的对象。用户在 Settings 中可以查看平台有哪些 Skill、管理员可以控制权限，但日常使用中 Skill 是透明的。

### 3.3 Channel（通道）

Channel 是 Cowork 与外部世界的双向连接——产出物通过 Channel 发布到该去的地方，外部事件通过 Channel 触发 Cowork 工作。

**Channel 的两个方向：**

**输出**——App 的结果发布到外部：
- 报告 → 邮件发送给管理层
- 告警 → 飞书群 / 钉钉群推送
- 周报 → 归档到 Confluence
- 数据 → 写入到数据库或 API

**输入**——外部事件触发 App 运行：
- 飞书群里有人 @bot 提问 → 触发对应 App
- 收到特定类型的客户邮件 → 自动分析
- Webhook 收到系统告警 → 触发分析 App
- 定时调度 → 每周一 9:00 自动运行

**Channel 类型：**
- IM（飞书、钉钉、Slack、Teams）— 双向：可推送消息，也可接收 @mention 或指令
- 邮件 — 双向：可发送，也可监听特定邮箱的来信
- 文档平台（Confluence、Notion）— 主要输出：归档报告和文档
- Webhook / API — 双向：接收外部系统事件，也可回调外部系统
- 定时调度 — 输入：按时间点触发 App 运行

**Channel 与 Knowledge 的区别：**
- Knowledge 是 AI 的参考资料库（被动的，AI 去查）——包括用户文档和已连接的数据源（Source）
- Channel 是 Cowork 与外部世界的交互通道（主动的，推送出去、接收进来）

一个外部系统可以同时作为 Knowledge 的数据源和 Channel。比如飞书：作为数据源，AI 可以从中读取历史消息作为参考；作为 Channel，App 可以向飞书群推送结果，也可以接收飞书群中的指令触发运行。用户首次连接飞书时，系统同时在 Knowledge 和 Channels 中配置好，用户不需要连两次。

### 3.4 App（应用）— 核心概念

App 是用户在 Cowork 中构建的可复用应用——面向特定问题的完整解决方案。

**一个 App 不是 Agent 编排图，而是一组意图和约束的声明：**

- **目标**：要达成什么（"分析各品类销售异常并生成简报"）
- **数据范围**：使用哪些数据源、什么时间范围、什么筛选条件
- **质量标准**：自然语言描述的判断准则（"关注环比下降超过 15% 的品类，但排除季节性因素"）
- **输出要求**：产出物的类型、格式、风格、受众（"面向管理层的简报，结论先行，每个品类分析不超过 3 句话"）
- **输出通道**：产出物发布到哪些 Channel（邮件、飞书群、Confluence）
- **触发方式**：什么情况下运行（手动、定时调度、外部 Channel 事件）
- **信任边界**：哪些环节需要人审阅后才能继续，哪些可以全自动
- **参数**：每次运行可调节的变量（日期范围、产品线等）

**App 运行时，AI 自动调用 Knowledge 和 Skill：** App 定义中的数据范围指定了显式依赖的数据源，AI 在执行时自动从 Knowledge 中调用相关上下文（比如用户的偏好和过去的分析结论），并自主选择需要的 Skill 来完成任务。同一个 App，不同用户运行可能产出不同风格的结果，因为每个人的 Knowledge 不同。

**关键设计原则：** 用户定义"做什么、做到什么标准"，不定义"怎么做、拆几步"。AI 在运行时自行决定调用哪些 Skill、以什么顺序执行。这意味着当平台新增 Skill 或 AI 能力升级时，同一个 App 会自动产出更好的结果，用户不需要重新编排。

**App 的可观测性：** 运行时系统展示 AI 的执行过程——用了哪些 Skill、访问了哪些数据、引用了哪些知识库资料、做出了什么判断。这是透明度和信任的基础，但它是"运行后可查看的日志"，不是"运行前需要设计的蓝图"。

### 3.5 Artifact（产出物）

App 或即时任务的产出物。有类型、有版本、可追溯到数据来源。

**Artifact 类型：**
- 文档报告（富文本、含图表）
- 结构化数据（表格、JSON）
- 邮件
- IM 消息（飞书、钉钉、Slack）
- 演示文稿
- 待办事项 / 行动清单

**Artifact 的两个去向：**
1. **通过 Channel 发布到外部** — 报告发邮件、告警推飞书群、文档归档到 Confluence
2. **自动回流到 Knowledge** — 每次产出都自动保存到知识库，成为 AI 未来的参考

第一次写周报时 AI 可能需要很多指导，但第十次时 AI 已经知道你的风格、常用的分析角度、偏好的措辞。这个积累是自动的，用户无需操作。

### 3.6 渐进式固化：从一次性使用到组织工具

App 不是一次性创建的，而是沿着一个自然路径渐进生成的：

```
Level 0 — 即时执行
  用户在 Command Bar 说"帮我看看上周销售异常"
  AI 调用相关 Skill 执行，产出结果
  → 结果可直接使用，执行记录自动保存

Level 1 — 个人 App
  用户觉得这个分析有价值，点击"保存为 App"
  系统从执行记录中提取意图和约束，生成 App 草稿
  → 下次可以一键重新运行

Level 2 — 参数化 App
  用户希望灵活调整，在 App 中标记哪些是可变参数
  → 每次运行可以调整日期范围、产品线等

Level 3 — 共享 App
  用户把 App 分享给同事，设置可见范围
  → 同事可以直接使用，无需了解底层逻辑

Level 4 — 组织 App
  App 被发布为正式应用，配置 Channel 触发（定时/事件驱动）和输出通道
  → 成为组织基础设施的一部分
```

每一步都是上一步的自然延伸，不需要"重新创建"或"切换模式"。

## 4. 产品原则

### 4.1 AI 自动使用你的资料

用户把文档和数据源放进知识库，AI 干活时自动找到相关的来用。用户不需要每次指定"用哪个文件"——AI 自己判断。用的不对？说一句就改。用 Cowork 产出的内容自动成为未来的参考资料，越用积累越多，AI 起点越高。

### 4.2 用户定义意图和约束，AI 决定执行方式

不要求用户理解 Agent 编排或选择 Skill。用户说"做什么"和"做到什么程度"，AI 自己决定用哪些 Skill、调用哪些知识和数据。当平台新增 Skill 或 AI 能力提升时，同样的 App 定义自动产出更好的结果。

### 4.3 渐进式固化，而非模式切换

不存在"使用模式"和"创建模式"的切换。每次使用都可能自然延伸为一个可复用的 App。保存、参数化、共享、发布——每一步都是轻量操作，不是重新学习一套新界面。

### 4.4 自然语言是主要交互方式

定义 App、调整参数、运行任务、修改输出——都通过自然语言完成。表单和结构化控件作为辅助存在（用于不知道说什么时的引导），但不是主要交互。

### 4.5 人始终可以介入

AI 自动执行，但人决定信任边界。每个环节可配置为"自动执行"或"需审阅"。信任可以随时间建立后逐步放开。运行过程透明可观测——包括 AI 引用了哪些 Knowledge，这样用户能判断 AI 的判断依据是否合理。

### 4.6 产出物是一等公民

不是"AI 给了你一段文本你自己去排版"。产出物有类型、有格式、有分发渠道。报告就是报告的样子，邮件就是可以直接发送的邮件。

### 4.7 越用越懂你

三个维度同时在"越用越好"：Knowledge 让 AI 更懂你的上下文和偏好，App 随反馈持续优化让流程更精准，平台新增 Skill 让 AI 能力更强。三者叠加的效果是：Cowork 用得越久，每次任务启动时的起点越高。
